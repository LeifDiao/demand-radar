# 报告结构模板（答案优先）

报告读者是**提出需求的人**（创始人/PM），不是分析师。第一屏必须回答：「我的需求成不成立？为什么？」**核心是「判断需求」**——告诉用户这是个什么需求（真实/伪、红海/蓝海、有没有楔子），而不是堆数据、也不是急着教怎么做。分析过程和证据收到最后折叠。报告由结构化 `report.json` 驱动，`generate_report.py --report` 渲染。

视觉是信号网格 / 蓝图风（与落地页同家族）：纸白网格底 + 墨黑 + 信号绿，Space Grotesk + IBM Plex Mono。措辞要配得上这套冷静、精确的仪器感——不煽情、不堆术语。

## 呈现顺序（generate_report.py 固定按此渲染）

1. **结论**（最重要，拆两段）
   - ① **裁决 hero**：你问的需求(`one_liner`) → 一句话判断(`verdict_sub`) + 一句话需求状态/方向(`verdict_sub2`) → 需求判定 pill(`demand_tags` + 置信度) → 右侧大分数 `40/100` + 裁决 → 底部「这分数怎么读」刻度条（四档 + 指针，自动按 `weighted_pct` 画）
   - ② **为什么是这个结论**：`why` 卡片（与「该怎么做」同款 UI），每条标 **加分/减分/注意**，让读者看懂分数怎么来的。最少 5 条，这是核心，宁多勿少。
2. **评分卡**：7 维归 3 类（需求真实性 / 商业潜力 / 竞争格局）带进度条
3. **该怎么做**：`actions` 可执行下一步（次要部分，不用太多）
4. **详情**：`voices` 不同领域怎么说（按来源汇总，大领域标签）+ 双支柱对账 + 红队 + 替代方案地图
5. **分析流程**：`method` 方法统计 + 流程；`evidence` 全量证据**默认折叠**（大家基本只看结果）

## 写作要点

- **结论是灵魂，先判断需求**。`verdict_sub` 第一句就给判断（痛点真不真、方向对不对）；`verdict_sub2` 说清需求状态或该往哪转——是「判断需求」，不是教怎么做。两句都要短、人话、可加粗。
- **`demand_tags` 用固定词表**（见 validation-framework.md F 节）：需求真实度（伪/弱/真实/强刚需）+ 竞争状态（蓝海/有空间/红海·已饱和）+ 机会位置（有楔子/仅细分有缝/无楔子）。`aggregate_scores.py` 的 `demand_state` 给前两类参考，竞争状态按竞品证据补。kind 决定颜色：good 绿 / bad 红 / warn 橙。
- **`why` 是核心，宁多勿少（≥5）**。每条 `kind` 用 `+`（加分）/`-`（减分）/`~`（注意），`title` 是结论点、`body` 挂事实。读者看完这组卡片就该明白「为什么是这个分」。致命项在 title 里标「· 致命」。
- **`voices` 按证据来源动态归类**：有几类来源就写几张（社区/应用评论/报告/资本/搜索/社交…），`domain` 写来源领域名（脚本按关键词配图标配色），`lean` 标这个领域整体倾向支撑/反对/矛盾，`summary` 一句话概括它怎么看这个需求。
- **`actions` 是次要部分**：可执行（换 ICP / 落地页预售 / 收窄场景），动词开头，别写「继续观察」这种废话。不用堆太多，2-3 条即可。
- **scorecard 顺序固定**为上面 7 维（脚本按 0-2/3-4/5-6 切成需求真实性/商业潜力/竞争格局三类），第 6 维是「差异化楔子」不是「竞争空白」。分数用闭环重评后的终评。
- **not_verified 必填**：把单源线索、只有意见证据、需付费源的点如实列出，不把「没查到」写成「不存在」。
- 每条 `why` / `voice` / `finding` 挂 `evidence_ids`，对应折叠的 evidence 附录可溯源。

## 措辞：去 AI 味（中文）

报告读者是要拿它做决定的人，写得像一个直率的同行在跟你说话，不是模型在介绍自己。

- **下判断，别和稀泥**。「数据不足以支撑这是个普遍需求」＞「这个需求可能在某些场景下存在一定潜力」。
- **删掉 AI 腔套话**：赋能、打造、助力、一键、闭环（指业务时）、值得注意的是、综上所述、在……的同时、不仅……而且。
- **少用排比与「首先/其次/再者」**的流水账结构；用具体事实代替修饰词。
- **数字和出处优先于形容词**。「App Store 84953 条评分里反复出现『太复杂』」＞「用户普遍反映体验不佳」。
- 好坏对照：
  - ✗「Notion 作为一款功能强大的生产力工具，其复杂性可能对部分用户构成了一定的使用门槛。」
  - ✓「真痛点是 Notion 太复杂——但抱怨的人多是免费个人用户，不是会掏钱的买家。需求真，买家错。」

## 语气红线（见 forbidden-patterns.md）

不用 AI 陈词；不把推测写成事实；不为了让用户高兴抬高裁决；低证据就给低置信度。
