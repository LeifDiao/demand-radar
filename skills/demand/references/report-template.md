# 报告结构模板（答案优先）

报告读者是**提出需求的人**（创始人/PM），不是分析师。第一屏必须回答：「我的需求解决了没？我该干嘛？」分析过程和数据收到最后折叠。报告由结构化 `report.json` 驱动，`generate_report.py --report` 渲染。

视觉是信号网格 / 蓝图风（与落地页同家族）：纸白网格底 + 墨黑 + 信号绿，Space Grotesk + IBM Plex Mono。措辞要配得上这套冷静、精确的仪器感——不煽情、不堆术语。

## 呈现顺序（generate_report.py 固定按此渲染）

1. **结论 hero**：你问的需求 → 裁决（带加权得分 + 靶角锁定）→ 大白话答案 → 关键判断 `tldr_points` → 量化读数条（agent 数 / 证据数 / 已验证信号 / 轮次，自动从 `method` 取）
2. **该怎么做**：建设性下一步（对用户最有用）
3. **评分卡**：7 维归 3 类（需求真实性 / 商业潜力 / 竞争格局）带进度条
4. **详情**：关键发现 / 双支柱对账 / 红队 / 替代方案地图 / 未验证
5. **分析流程 & 数据**：方法统计 + 全量证据附录（默认折叠）

## report.json 字段

```json
{
  "question": "更简单的 Notion",
  "one_liner": "一句话复述用户想做的东西",
  "verdict": "Go | 有条件做 | 转向 | 别做",
  "confidence": "高 | 中 | 低",
  "answer": "面向提问者的大白话结论：需求解决了没 + 为什么（2-4句，能看懂，可用 **加粗**）",
  "tldr_points": ["[+] 正面判断", "[-] 负面判断", "[~] 中性/转向判断"],
  "weighted_pct": 40.0,
  "actions": [
    {"title": "下一步标题", "priority": "高|中|低", "detail": "具体可执行的建议"}
  ],
  "scorecard": [
    {"dim": "痛点强度", "score": 2, "max": 5, "weight": 1.5, "confidence": "中", "note": "一句话依据"}
  ],
  "findings": [
    {"title": "关键发现标题", "body": "正文", "evidence_ids": ["A7","B1"]}
  ],
  "contradictions": "双支柱对账：自上而下 vs 自下而上是否一致，矛盾要点出",
  "red_team": "红队最致命弱点 + 它把裁决拉到哪",
  "alternatives": [{"current": "用户现在怎么凑合", "gap": "缺口/机会点"}],
  "not_verified": ["未能验证的点1", "需付费源才拿得到的点2"],
  "method": {
    "agents": 8, "evidence_count": 38, "signals_verified": 10, "rounds": 1,
    "sources": ["Hacker News", "App Store 评论", "行业报告"]
  },
  "evidence": [
    {"id":"A1","platform":"HN","source_type":"硬|行为|意见|分析师","claim":"主张","quote":"原话","url":"...","date":"2025-05"}
  ]

```

## 写作要点

- **answer** 是整份报告的灵魂：用人话说清「解决了没、为什么」，给方向不堆术语。开门见山，第一句就给判断，再补一两句最关键的理由。
- **actions** 决定报告有没有用：每条要可执行（换 ICP / 做落地页预售 / 收窄场景），写成动词开头的祈使句。不要「继续观察」「深入调研」这种废话。
- **tldr_points** 用 `[+]`（正面）`[-]`（负面）`[~]`（中性/转向）开头，渲染成 ✓/✗/→。每条是一个具体判断，不是话题标签。
- **scorecard** 分数用闭环重评后的终评，note 一句话挂依据。
- **not_verified 必填**：把单源线索、只有意见证据、需付费源的点如实列出，不把「没查到」写成「不存在」。
- 每条 finding 挂 `evidence_ids`，对应 evidence 附录可溯源。

## 措辞：去 AI 味（中文）

报告读者是要拿它做决定的人，写得像一个直率的同行在跟你说话，不是模型在介绍自己。

- **下判断，别和稀泥**。「数据不足以支撑这是个普遍需求」＞「这个需求可能在某些场景下存在一定潜力」。
- **删掉 AI 腔套话**：赋能、打造、助力、一键、闭环（指业务时）、值得注意的是、综上所述、在……的同时、不仅……而且。
- **少用排比与「首先/其次/再者」**的流水账结构；用具体事实代替修饰词。
- **数字和出处优先于形容词**。「App Store 84953 条评分里反复出现『太复杂』」＞「用户普遍反映体验不佳」。
- 好坏对照：
  - ✗「Notion 作为一款功能强大的生产力工具，其复杂性可能对部分用户构成了一定的使用门槛。」
  - ✓「真痛点是 Notion 太复杂——但抱怨的人多是免费个人用户，不是会掏钱的买家。需求真，买家错。」

## 语气红线（见 forbidden-patterns.md）

不用 AI 陈词；不把推测写成事实；不为了让用户高兴抬高裁决；低证据就给低置信度。
