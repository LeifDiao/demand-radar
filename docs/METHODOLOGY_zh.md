# Demand Radar 方法论

Demand Radar 的报告质量约 90% 来自框架，只有 10% 来自爬了多少数据。本页讲框架。

## 1. 框定（化用 The Mom Test）

任何搜索之前，先把模糊想法逼成**可证伪假设**，含五部分：ICP · 要解决的 Job/痛点 · 现状凑合 · 待验断言 · Go 判据。好的断言里有能被数据打脸的东西（人群 + 行为 + 动机/付费）。坏断言（「这市场很大」）无法被任何证据推翻，直接打回。

## 2. 两根证据支柱

| 支柱 | 数据源 | 回答 |
|---|---|---|
| **自上而下** | 行业报告（Gartner/IDC/Statista）、官方统计、融资数据、上市公司文件、政策 | 市场规模、增长、资本流向、官方口径 |
| **自下而上** | Hacker News、App Store / Google Play 评论、搜索需求、社交 | 痛点真不真、谁有、现在用什么凑合、付费意愿 |

两边交叉验证。**矛盾本身就是头等发现**——「报告说市场大但社区零讨论」要么是沉默的 to B 市场，要么是被分析师吹大的伪市场。

## 3. 证据分级

| 级别 | 定义 | 能支撑 |
|---|---|---|
| **硬证据** | 带钱的真实行为：付费、留存、融资、财报 | 付费意愿 / 市场规模结论 |
| **行为证据** | 真实使用/搜索/讨论：评论原话、搜索趋势、高赞抱怨 | 痛点真实 / 普遍性 |
| **意见证据** | 「我会买」「听起来不错」 | 只能当线索，不能单独成结论 |
| **分析师观点** | 报告里的预测/定性判断 | 给方向，需与行为证据交叉 |

## 4. 三角验证（`triangulate.py`）

一个信号只有被 ≥2 个*独立*源确认才算「已验证」（独立 = 不同平台、非互相转载）。
- 1 源 → 线索（待验证）
- 2 源 → 成立（中）
- ≥3 源 **且**含硬证据 → 强成立（高）

模型给每条证据打 `signal` 标签（语义判断），脚本数独立源、定置信度（确定性）。防止单源线索被嘴软说成「已验证」。

## 5. 7 项评分卡 + 多评委打分

痛点强度（×1.5）· 普遍性 · 现有替代缺口 · **付费意愿（×2）** · 市场规模与趋势 · 竞争/空白 · 可触达性。每项 0–5 分。

3 个评委 agent（中立 / 乐观 / 严格）独立打分；`aggregate_scores.py` 取每维中位数、标出评委分歧 ≥2 的维度、算加权裁决档：

| 加权占比 | 裁决 |
|---|---|
| ≥ 70% | 可以做 |
| 50–70% | 有条件做 |
| 30–50% | 转向 |
| < 30% | 别做 |

## 6. 对抗红队 + 闭环重评

专门的红队 agent 用七种杀法攻击假设（伪需求 / 维生素非止痛药 / 已饱和 / 在萎缩 / 没人付钱 / 触达不了 / 样本偏误），反证也挂引用。有数据支撑的攻击会被纳入并**重跑聚合**得最终裁决。任何「可以做」都要扛住它。红队找不到反证时，「未能证伪」本身就是强正面信号。

## 7. 诚实红线

- 不编造数据——每个数字都能追到 URL
- 无证据不下结论——缺数据标「数据不足」，不靠猜
- 「没查到」绝不写成「不存在」——进强制的**「未能验证」**一节
- 置信度逐条标注，绝不为讨好用户抬高

## 零登录数据获取

Demand Radar **只用免费官方 API + 网页搜索**——不写爬虫、无 OAuth、不要 key。Reddit 因屏蔽爬虫被排除在默认运行外；自下而上的社区信号由 Hacker News、App Store / Google Play 评论、以及对比博客与可访问论坛的普通搜索覆盖。需要登录或反爬的源（大规模 Reddit、Product Hunt token、G2/Amazon 全量评论、精确搜索量）留给未来的付费可插拔连接器层。
